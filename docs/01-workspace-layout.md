# docs/01-workspace-layout.md

> The on-disk format for a Mike workspace. This is the source of truth; SQLite is a derived index.

## Top-level directory tree

```
<workspace>/                               ← user-picked folder, portable
│
├── .mike/                                 ← derivable index + runtime state
│   ├── workspace.json                     ← {schema_version, id, created_at}
│   ├── mike.db                            ← SQLCipher-encrypted SQLite
│   ├── secrets.enc                        ← AES-256-GCM secrets bundle
│   ├── secrets.salt                       ← Argon2id salt
│   ├── runtime/                           ← lifecycle, not data
│   │   ├── workspace.lock                 ← single-instance lock
│   │   ├── backend.json                   ← {port, pid, started_at}
│   │   └── sidecars/
│   │       ├── docling.json
│   │       └── eyecite.json
│   ├── sidecar-cache/                     ← model weights, never user data
│   │   └── docling/
│   ├── certs/                             ← self-signed for HTTPS:3002
│   │   ├── server.key
│   │   └── server.crt
│   └── logs/                              ← rotating, append-only where supported
│       ├── backend.log
│       ├── audit.log
│       ├── electron.log
│       └── sidecar-docling.log
│
├── matters/                               ← human-readable archive (the data)
│   ├── _unfiled/                          ← items pending matter assignment
│   │   ├── matter.md                      ← always present, isolation: shared
│   │   └── items/
│   │       └── <kind>-<ulid>.md
│   │
│   ├── <client-slug>/                     ← one folder per client
│   │   ├── client.md                      ← client metadata + freeform notes
│   │   │
│   │   └── <matter-slug>/                 ← one folder per matter
│   │       ├── matter.md                  ← matter metadata, isolation_mode
│   │       ├── items/                     ← cite-able content
│   │       │   ├── document-<ulid>.md
│   │       │   ├── email-<ulid>.md
│   │       │   ├── note-<ulid>.md
│   │       │   ├── appointment-<ulid>.md
│   │       │   ├── contact-<ulid>.md
│   │       │   └── chat-<ulid>.md
│   │       ├── attachments/               ← original binaries, sha256-named
│   │       │   ├── <sha256-prefix>.pdf
│   │       │   └── <sha256-prefix>.docx
│   │       └── exports/                   ← drafts, .mikeprj outputs
│   │           ├── draft-2026-05-11.docx
│   │           └── stevens-handoff.mikeprj
│   │
│   └── ... more clients
│
└── corpora/                               ← optional public corpora (defer)
    └── <corpus-name>/
        └── <doc-id>.md
```

## What's source-of-truth vs derived

| Location | Source of truth | Derivable |
|---|---|---|
| `<workspace>/matters/**/*.md` | ✅ The data | — |
| `<workspace>/matters/**/attachments/*` | ✅ Original binaries | — |
| `<workspace>/.mike/mike.db` | — | ✅ From `.md` files via `db::rebuild` |
| `<workspace>/.mike/secrets.enc` | ✅ Secrets (no other copy) | — |
| `<workspace>/.mike/sidecar-cache/` | — | ✅ Re-downloads on demand |
| `<workspace>/.mike/certs/` | — | ✅ Regenerated on first launch |
| `<workspace>/.mike/logs/` | — | Not regenerated, but disposable |

**Rule:** if `<workspace>/.mike/` is deleted, restarting Mike (after PIN entry) rebuilds it. If anything in `<workspace>/matters/` is deleted, that data is lost. Document this prominently in user-facing docs.

## Naming conventions

### Item IDs
**ULID** (Universally Unique Lexicographically Sortable Identifier — like UUID but sortable by creation time). 26 characters, base32. Generated by the backend.

Format: `<kind>-<ulid>.md`
Example: `email-01J5K3R0H9NMVE5T8ZQA3W7BJP.md`

The `kind` prefix is duplicated from the frontmatter for filesystem-level scanability — a lawyer browsing in Finder can tell what's what without opening files.

### Client and matter slugs
Lowercase, kebab-case, ASCII only, max 60 chars. Derived from the user-entered name:
1. Lowercase
2. Replace any non-alphanumeric run with a single `-`
3. Trim leading/trailing `-`
4. Truncate to 60
5. If collision in the parent directory, append `-<short-ulid>` (8 chars)

Examples:
- "Acme Corp." → `acme-corp`
- "Stevens Litigation (E.D. Cal.)" → `stevens-litigation-e-d-cal`
- Two clients both reducing to `smith` → `smith` and `smith-7zxyzpd2`

The slug is **not the display name**. The display name lives in `client.md` / `matter.md` frontmatter. The slug is a stable filesystem identifier.

### Attachment naming
Attachments are content-addressed: `<sha256-prefix-12>.<original-ext>`.

Example: `84a2b97fc1e3.pdf` (the first 12 hex chars of the SHA-256 of the file bytes).

12 chars is 48 bits of entropy — collision probability for any realistic workspace is negligible. We keep the original extension so the file opens with the right app when double-clicked.

The full SHA-256 lives in `items[*].frontmatter.attachments[*].sha256` and in the SQLite `attachments` table for ref-counting and integrity checks.

## File ordering and atomicity

All `.md` writes follow:
1. Write content to `<target>.tmp`.
2. `fsync` the file.
3. `rename(<target>.tmp, <target>)`. POSIX rename is atomic on the same filesystem.

A crash between (1) and (3) leaves the previous file intact and an orphan `.tmp`. Startup repair removes any `.tmp` files older than 60s.

## Two-phase write coordination

When updating an item:
1. Compute the new `content_hash` and frontmatter.
2. Write the `.md` atomically (above).
3. Begin SQLite transaction: update `items` row, regenerate chunks, update FTS + vec0.
4. Commit transaction.

If (3) fails, the `.md` is ahead of SQLite. Startup repair detects this via random-sample content_hash mismatch and re-indexes affected items.

If (2) fails, SQLite is unchanged; the user retries.

## Filesystem watcher

The backend watches `<workspace>/matters/` with the `notify` crate. Behavior:

- **Self-write suppression:** an in-memory `HashSet<PathBuf>` tracks paths the backend just wrote in the last 2 seconds (TTL via tokio `interval`). Events for those paths are ignored.
- **External edit:** read the file, compute body hash, compare to `items.content_hash`. Match → no-op. Mismatch → re-parse frontmatter, update SQLite row, re-chunk, re-embed, publish SSE.
- **External delete:** soft-delete the SQLite row (set `deleted_at`); preserve chunks for rollback. After 30 days, hard-delete.
- **External create:** if it has valid frontmatter, ingest as a new item. If not, log a warning and ignore.

Throttle events to one batch per directory per second to avoid storms during external bulk operations.

## `workspace.json`

```json
{
  "schema_version": 1,
  "id": "01J5K1RX...",
  "created_at": "2026-05-08T14:00:00Z",
  "mike_version_first": "0.1.0"
}
```

`id` is the workspace's ULID — used in audit logs to correlate cross-workspace operations.

Versioning rules in `docs/04-versioning.md`.

## Lock file

`<workspace>/.mike/runtime/workspace.lock`:

```json
{
  "pid": 8421,
  "hostname": "dave-macbook",
  "started_at": "2026-05-11T09:30:00Z",
  "mike_version": "0.1.0"
}
```

On startup:
1. If file exists, try to signal the listed pid (`kill -0` on Unix; `OpenProcess` on Windows). If alive, refuse to open with: "Workspace is in use by Mike on `<hostname>` (pid `<pid>`)."
2. If pid is gone, offer "Looks stale (last seen on `<hostname>`). Take over?"
3. On accept (or if no file), write a fresh lock file with our pid/hostname.
4. On clean exit, delete the lock file.
